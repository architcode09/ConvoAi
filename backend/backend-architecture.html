iMessage Backend Architecture

This HTML guide explains how the backend is structured, what each file does, and how requests flow through the app.

1. Big picture

The backend lives in backend/src. It is an Express server with three main responsibilities:

  • Authenticate and sync users through Clerk webhooks.
  • Handle chat operations like fetching conversations, sending messages, and uploading media.
  • Support real-time presence / message delivery via Socket.IO.
Frontend Browser or mobile client sends API calls and socket events.
↓
Express Server src/index.js
↓
MongoDB User and message data storage.
↓
Socket.IO Real-time online status and push messages.

2. Entry point: src/index.js

This file is the first code that runs when the backend starts.

What it does
  • Creates an Express app and HTTP server.
  • Mounts the Clerk webhook endpoint at /api/webhooks/clerk.
  • Loads JSON parsing, CORS, and Clerk authentication middleware.
  • Registers route groups for auth and message APIs.
  • Starts the HTTP server and connects to MongoDB.
  • Starts a periodic health ping in production.
app.use("/api/webhooks/clerk", express.raw({ type: "application/json" }), clerkWebhook); app.use(express.json()); app.use(cors({ origin: FRONTEND_URL, credentials: true })); app.use(clerkMiddleware()); app.use("/api/auth", authRoutes); app.use("/api/messages", messageRoutes); server.listen(PORT, () => { connectDB(); console.log("Server is up and running on PORT:", PORT); if (process.env.NODE_ENV === "production") job.start(); });

Triggered when: the backend process starts with npm start or npm run dev.

router.use(protectRoute); router.get("/users", getUsersForSidebar); router.get("/conversations", getConversationsForSidebar); router.get(":id", getMessages); router.post("/send/:id", upload.single("media"), sendMessage);

What each endpoint does:

  • GET /api/messages/users — returns other users for the sidebar.
  • GET /api/messages/conversations — returns chat partners in most recent order.
  • GET /api/messages/:id — returns all messages between the current user and the selected user.
  • POST /api/messages/send/:id — sends a new message to the selected user.

5.1 Message controller details

src/controllers/message.controller.js
export async function getUsersForSidebar(req, res) { const loggedInUserId = req.user._id; const filteredUsers = await User.find({ _id: { $ne: loggedInUserId } }).select("-clerkId"); res.status(200).json(filteredUsers); } export async function getConversationsForSidebar(req, res) { const conversations = await Message.aggregate([...]); res.status(200).json(conversations); } export async function sendMessage(req, res) { const { text } = req.body; const { id: receiverId } = req.params; const senderId = req.user._id; if (req.file) { ... uploadChatMedia(req.file); } const newMessage = new Message({ senderId, receiverId, text, image: imageUrl, video: videoUrl }); await newMessage.save(); const receiverSocketId = getReceiverSocketId(receiverId); if (receiverSocketId) io.to(receiverSocketId).emit("newMessage", newMessage); res.status(201).json(newMessage); }

Triggered when: the frontend requests chat data or sends a message.

Note: sendMessage can upload media if the request has a file, and then it sends a real-time socket event to the receiver if they are online.

6. Media upload helper

src/middleware/upload.middleware.js
export const upload = multer({ storage: multer.memoryStorage(), limits: { fileSize: MAX_FILE_SIZE }, fileFilter: (req, file, cb) => { const isImage = file.mimetype.startsWith("image/"); const isVideo = file.mimetype.startsWith("video/"); if (!isImage && !isVideo) cb(new Error("Only image and video uploads are allowed")); cb(null, true); }, });

This middleware reads file uploads into memory and only allows images or videos. It is used by the message send endpoint.

7. Real-time socket server

src/lib/socket.js
const io = new Server(server, { cors: { origin: [allowedOrigin] } }); const userSocketMap = {}; io.on("connection", (socket) => { const userId = socket.handshake.query.userId; if (userId) userSocketMap[userId] = socket.id; io.emit("getOnlineUsers", Object.keys(userSocketMap)); socket.on("disconnect", () => { ... }); }); export { app, server, io, getReceiverSocketId };

What it does:

  • Keeps track of connected users with userSocketMap.
  • Broadcasts online user IDs with getOnlineUsers.
  • Allows the message controller to push events to a specific receiver.

Triggered when: a client opens a socket connection and when a user disconnects.

src/models/user.model.js
const userSchema = new mongoose.Schema({ clerkId: { type: String, required: true, unique: true }, email: { type: String, required: true, unique: true }, fullName: { type: String, required: true }, profilePic: { type: String, default: "" }, }, { timestamps: true });
src/models/message.model.js
const messageSchema = new mongoose.Schema({ senderId: { type: mongoose.Schema.Types.ObjectId, ref: "User", required: true }, receiverId: { type: mongoose.Schema.Types.ObjectId, ref: "User", required: true }, text: { type: String }, image: { type: String }, video: { type: String }, }, { timestamps: true });

The two models connect users and messages. A message stores the sender, receiver, text, and optionally image or video URLs.

12. Environment variables used

  • PORT — backend server port.
  • FRONTEND_URL — frontend origin and health ping target.
  • MONGO_URI — MongoDB connection string.
  • IMAGEKIT_PRIVATE_KEY — required to upload media to ImageKit.
  • CLERK_WEBHOOK_SIGNING_SECRET — verifies Clerk webhook requests.

13. How the pieces are connected

Here is the runtime connection graph in simple terms:

  1. src/index.js starts Express, socket server, DB connection, and webhook handling.
  2. src/webhooks/clerk.webhook.js keeps user profiles synced from Clerk into MongoDB.
  3. src/routes/auth.route.js and src/routes/message.route.js expose API endpoints for the frontend.
  4. src/middleware/auth.middleware.js protects message routes and ensures the user exists in the local DB.
  5. src/controllers/message.controller.js performs all messaging operations and sends real-time updates.
  6. src/lib/socket.js tracks online users and delivers live message events.
Key idea: The backend is mainly a chat API plus user sync. It does not handle frontend UI. The frontend logs in with Clerk, then calls the backend for user lists, conversations, and sending messages. Webhooks keep the local user database updated.
