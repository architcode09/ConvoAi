How Signup, Clerk, Webhooks, and MongoDB Are Connected

This page explains the full wiring in the easiest way possible. The most important idea is: clicking SignUpButton does not directly save data into MongoDB. First the user is created in Clerk, then the app and webhook together help sync that user into your backend database.

1. UI starts the process

The user clicks the signup button in your frontend.

2. Clerk creates the auth user

Clerk handles signup and stores the user in Clerk's system.

3. Backend stores app data

Your backend writes the user into MongoDB so your app can use it later.

Big Picture First

After the user signs up, two useful sync paths can happen in your project:

  • The frontend notices the signed-in user and calls /api/users/sync.
  • Clerk also sends a server-to-server webhook to /api/webhooks/clerk.
Very important truth:

The user does not click a button that directly calls the webhook. The webhook is sent by Clerk's server, not by the browser. The browser talks to Clerk's UI and to your backend sync route.

Block Diagram: Full Signup Journey

This is the end-to-end path from the signup button all the way to the database.

1. User clicks Signup

In frontend/src/App.jsx, the user clicks <SignUpButton />. Clerk opens the signup modal on the UI.

→

2. Clerk handles signup

Clerk collects email, password, and profile details. Clerk is doing the authentication work here.

→

3. User is created in Clerk

Clerk stores the new user in its own system. This is why you can see the user in the Clerk dashboard.

After the user is created in Clerk, the flow splits into two paths:

Path A: Frontend Sync

The browser now knows the user is signed in. Your React app reads the Clerk user with useUser() and sends a POST request to /api/users/sync.

Path B: Clerk Webhook

Clerk's server sends an event like user.created to your backend route /api/webhooks/clerk.

4A. Frontend sends user data

The frontend sends clerkId, email, fullName, and profilePic to /api/users/sync.

→

5A. Backend sync route runs

In backend/src/index.js, your backend receives the request and runs User.findOneAndUpdate(..., { upsert: true }).

→

6A. MongoDB stores the user

MongoDB saves the record in your User collection with fields like clerkId, email, fullName, and profilePic.

4B. Clerk sends webhook event

Clerk sends signed event data to /api/webhooks/clerk. This is server-to-server communication.

→

5B. Backend verifies the webhook

In backend/src/webhooks/clerk.webhook.js, your backend verifies the signature, trusts the event, and then saves or updates the user.

→

6B. MongoDB stays in sync

MongoDB gets updated again if needed. This is very useful for user.updated and user.deleted events too.

Connect Every Dot Step by Step

1

Frontend loads Clerk

In frontend/src/main.jsx, ClerkProvider is added. This gives your app access to Clerk auth features like signup, signin, and current user data.

2

User clicks Sign Up

In frontend/src/App.jsx, SignUpButton opens the Clerk signup modal. The browser is now talking to Clerk's auth UI.

3

Clerk creates the real auth user

Clerk stores the new user in Clerk's own system. That is why the user appears in the Clerk dashboard first.

4

The browser gets the signed-in user

After signup succeeds, useUser() in App.jsx can read the user object such as id, email, name, and image.

5

Frontend sends data to your backend

Your app calls /api/users/sync with the Clerk user details. This is your own backend API route.

6

Backend is already connected to MongoDB

In backend/src/lib/db.js, connectionDB() connects the server to MongoDB when the backend starts.

7

Backend writes the user

In backend/src/index.js, the sync route uses findOneAndUpdate with upsert: true. That means:

  • if the user already exists, update it
  • if the user does not exist, create it
8

Clerk also sends a webhook

Separately, Clerk sends a webhook event like user.created or user.updated to your backend. This is not from the browser. This comes from Clerk's server.

9

Backend verifies the webhook safely

Your backend reads the raw body, checks the Clerk signing secret, and only then trusts the webhook data.

10

MongoDB stays in sync with Clerk

The webhook path helps keep your local database updated even when something changes later in Clerk, like a profile update or user deletion.

Why Use Both Frontend Sync and Webhook?

Using both is actually helpful in this app:

  • Frontend sync gives you a quick save right after the user signs up and lands in the app.
  • Webhook sync keeps your database correct even when the change comes from Clerk later.

So the frontend path is like an immediate save, and the webhook path is like a trusted background sync from Clerk.

Which File Does What?

frontend/src/main.jsx

Starts Clerk in the React app using ClerkProvider.

frontend/src/App.jsx

Shows the signup button and, after signup, reads the user and calls /api/users/sync.

backend/src/index.js

Starts Express, connects middleware, adds /api/users/sync, and mounts /api/webhooks/clerk.

backend/src/webhooks/clerk.webhook.js

Receives Clerk webhook events, verifies them, and updates MongoDB safely.

backend/src/models/user.model.js

Defines the user data shape stored in MongoDB: clerkId, email, fullName, and profilePic.

backend/src/lib/db.js

Creates the MongoDB connection so the backend can read and write users.

One-Line Simple Explanation

The user clicks signup in the UI, Clerk creates and stores the auth user, your app then reads that Clerk user, sends the details to your backend, and your backend stores the user in MongoDB while the Clerk webhook keeps everything synced safely in the background.
